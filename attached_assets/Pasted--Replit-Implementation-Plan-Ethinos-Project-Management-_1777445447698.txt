# Replit Implementation Plan: Ethinos Project Management Performance + Security Fixes

## Purpose

Use this document as a Replit Agent implementation brief. The goal is to apply the highest-impact backend, Firebase, scheduler, and frontend performance fixes in a safe order.

Repository:

```txt
https://github.com/theodorehayes86-source/Ethinos_Project_Management
```

Primary focus:

1. Fix critical security and correctness issues first.
2. Reduce full-database scans and whole-array writes.
3. Prevent duplicate scheduled jobs in multi-instance deployments.
4. Improve frontend rendering performance in the large task view.
5. Keep changes incremental and backward-compatible where possible.

---

## Assumptions

The codebase appears to use:

* React / Vite frontend under `artifacts/pmt-app`
* Express / TypeScript API under `artifacts/api-server`
* Firebase Realtime Database as the main operational datastore
* Firebase Admin SDK on the backend
* Microsoft Graph for email delivery
* Scheduled jobs for reminders, repeat tasks, weekly digests, and Keka sync

Do not rewrite the entire application. Apply changes in small, testable commits.

# Phase 1 — Critical Security + Correctness Fixes

## Task 1.1 — Lock down Firebase Realtime Database rules

### Problem

Current `database.rules.json` grants broad read/write access to authenticated `@ethinos.com` users. This lets any valid domain user bypass frontend/API role checks and directly modify sensitive data.

### File

```txt
database.rules.json
```

### Required change

Replace top-level global read/write rules with path-level rules. At minimum:

* Deny all by default.
* Allow users to read their own profile.
* Allow Admin / Super Admin users to manage users and settings.
* Restrict task/client data by explicit access paths.
* Add indexes for common lookups.

### Suggested baseline

```json
{
  "rules": {
    ".read": false,
    ".write": false,

    "roles": {
      "$uid": {
        ".read": "auth != null && (auth.uid === $uid || root.child('roles').child(auth.uid).val() === 'Admin' || root.child('roles').child(auth.uid).val() === 'Super Admin')",
        ".write": "auth != null && root.child('roles').child(auth.uid).val() === 'Super Admin'"
      }
    },

    "users": {
      ".indexOn": ["email", "managerId"],
      "$uid": {
        ".read": "auth != null && (auth.uid === $uid || root.child('roles').child(auth.uid).val() === 'Admin' || root.child('roles').child(auth.uid).val() === 'Super Admin')",
        ".write": "auth != null && (root.child('roles').child(auth.uid).val() === 'Admin' || root.child('roles').child(auth.uid).val() === 'Super Admin')",
        ".validate": "newData.hasChildren(['email', 'role'])"
      }
    },

    "settings": {
      ".read": "auth != null",
      ".write": "auth != null && (root.child('roles').child(auth.uid).val() === 'Admin' || root.child('roles').child(auth.uid).val() === 'Super Admin')"
    },

    "clientLogs": {
      "$clientId": {
        "$taskId": {
          ".read": "auth != null && root.child('clientAccess').child(auth.uid).child($clientId).val() === true",
          ".write": "auth != null && root.child('clientWriters').child(auth.uid).child($clientId).val() === true"
        }
      }
    },

    "tasksByAssignee": {
      "$uid": {
        ".read": "auth != null && (auth.uid === $uid || root.child('roles').child(auth.uid).val() === 'Admin' || root.child('roles').child(auth.uid).val() === 'Super Admin')",
        ".write": "auth != null && (root.child('roles').child(auth.uid).val() === 'Admin' || root.child('roles').child(auth.uid).val() === 'Super Admin')"
      }
    },

    "tasksByDueDate": {
      "$date": {
        ".read": "auth != null",
        ".write": "auth != null && (root.child('roles').child(auth.uid).val() === 'Admin' || root.child('roles').child(auth.uid).val() === 'Super Admin')"
      }
    },

    "schedulerLocks": {
      ".read": false,
      ".write": false
    },

    "rateLimits": {
      ".read": false,
      ".write": false
    }
  }
}
```

### Acceptance criteria

* Top-level `.read` and `.write` are both `false`.
* Non-admin users cannot write `settings`, `users`, scheduler state, or rate limit state.
* Task access is no longer globally open to every authenticated domain user.
* Existing app flows still work for authorized users.

### Files

```txt

artifacts/api-server/src/routes/auth.ts
```

### Required change

Use `lib/firebase-admin.ts` as the only Firebase Admin bootstrap module. Do not initialize Admin SDK directly inside route files.

### Target API

```ts
// artifacts/api-server/src/lib/firebase-admin.ts
import admin from "firebase-admin";

let app: admin.app.App | null = null;

export function getFirebaseAdmin(): admin.app.App {
  if (app) return app;

  if (admin.apps.length > 0) {
    app = admin.app();
    return app;
  }

  app = admin.initializeApp({
    credential: admin.credential.cert({
      projectId: process.env.FIREBASE_PROJECT_ID,
      clientEmail: process.env.FIREBASE_CLIENT_EMAIL,
      privateKey: process.env.FIREBASE_PRIVATE_KEY?.replace(/\\n/g, "\n")
    }),
    databaseURL: process.env.FIREBASE_DATABASE_URL
  });

  return app;
}

export function getAdminDatabase(): admin.database.Database {
  return getFirebaseAdmin().database();
}

export function getAdminAuth(): admin.auth.Auth {
  return getFirebaseAdmin().auth();
}
```

### Update imports

### Acceptance criteria

* No route file calls `admin.initializeApp()` directly.

## Task 1.4 — Make `/notify` server-authoritative

### Problem

The notify route accepts client-provided notification content and recipient data. Authenticated users should not be able to directly control email recipients or message bodies.

### File

```txt
artifacts/api-server/src/routes/notify.ts
```

### Required change

Change `/notify` so the client sends only identifiers. The server must load the task/user data, verify permissions, derive recipients, and generate email content internally.

### Target request body

```ts
type NotifyRequest = {
  type: "task-assigned" | "task-due" | "approval-required";
  clientId: string;
  taskId: string;
};
```

### Implementation sketch

```ts
function parseNotifyRequest(body: unknown): NotifyRequest {
  if (!body || typeof body !== "object") {
    throw new Error("Invalid request body");
  }

  const value = body as Partial<NotifyRequest>;

  if (
    value.type !== "task-assigned" &&
    value.type !== "task-due" &&
    value.type !== "approval-required"
  ) {
    throw new Error("Invalid notification type");
  }

  if (!value.clientId || !value.taskId) {
    throw new Error("Missing task reference");
  }

  return {
    type: value.type,
    clientId: value.clientId,
    taskId: value.taskId
  };
}

router.post("/notify", requireFirebaseAuth, async (req, res) => {
  const decoded = req.user!;
  const { type, clientId, taskId } = parseNotifyRequest(req.body);

  const task = await readFirebasePath<Task>(`clientLogs/${clientId}/${taskId}`);

  if (!task) {
    return res.status(404).json({ error: "Task not found" });
  }

  await assertCanNotify(decoded.uid, task);

  const assignee = await readFirebasePath<User>(`users/${task.assigneeId}`);

  if (!assignee?.email?.endsWith("@ethinos.com")) {
    return res.status(400).json({ error: "Invalid recipient" });
  }

  await sendTaskNotification({
    type,
    task,
    recipientEmail: assignee.email
  });

  return res.json({ ok: true });
});
```

### Acceptance criteria

* Request body no longer accepts arbitrary `to`, `subject`, `html`, or recipient email fields.
* The server verifies the authenticated user can trigger the notification.
* Email body and recipients are derived server-side.
* Invalid task IDs return 404.
* Unauthorized users return 403.

# Phase 2 — Scheduler Safety + Performance

## Task 2.1 — Add distributed scheduler locks

### Problem

Schedulers are started from the API server boot path. If multiple app instances run, each instance executes the same jobs, causing duplicate reminders, duplicate repeat tasks, duplicate digests, and duplicate external writes.

### Files

```txt
artifacts/api-server/src/index.ts
artifacts/api-server/src/lib/reminder-scheduler.ts
artifacts/api-server/src/lib/repeat-scheduler.ts
artifacts/api-server/src/lib/weekly-digest-scheduler.ts
artifacts/api-server/src/lib/keka-scheduler.ts
```

### Required change

Add a reusable Firebase-backed distributed lock utility and wrap every scheduled job run.

### New file

```txt
artifacts/api-server/src/lib/job-lock.ts
```

### Implementation

```ts
import { getAdminDatabase } from "./firebase-admin";

export async function withJobLock(
  name: string,
  ttlMs: number,
  job: () => Promise<void>
): Promise<void> {
  const db = getAdminDatabase();
  const lockRef = db.ref(`schedulerLocks/${name}`);
  const now = Date.now();
  const owner = `${process.env.HOSTNAME ?? "local"}:${process.pid}`;

  const result = await lockRef.transaction((current) => {
    if (current?.expiresAt && current.expiresAt > now) {
      return;
    }

    return {
      owner,
      startedAt: now,
      expiresAt: now + ttlMs
    };
  });

  if (!result.committed) return;

  try {
    await job();
  } finally {
    const snapshot = await lockRef.once("value");
    if (snapshot.val()?.owner === owner) {
      await lockRef.remove();
    }
  }
}
```

### Usage example

```ts
import { withJobLock } from "./job-lock";

cron.schedule("0 * * * *", () =>
  withJobLock("reminders-hourly", 55 * 60_000, runScheduledReminderChecks)
);
```

### Acceptance criteria

* Every scheduler job is protected by a named lock.
* Lock TTL is shorter than the schedule interval.
* Duplicate app instances do not run the same job concurrently.
* Failed jobs release locks or allow expiration.

## Task 2.2 — Avoid `users.find()` inside task loops

### Problem

Schedulers repeatedly scan the full users array for each task. This creates `O(tasks × users)` work.

### File

```txt
artifacts/api-server/src/lib/reminder-scheduler.ts
```

### Required change

Build `Map` indexes once per scheduler run.

### Before

```ts
const assigneeEmail =
  task.assigneeEmail ||
  users.find((u) => String(u.id) === String(task.assigneeId))?.email;
```

### After

```ts
const userById = new Map(users.map((user) => [String(user.id), user]));

const assigneeEmail =
  task.assigneeEmail ||
  userById.get(String(task.assigneeId))?.email;
```

### Acceptance criteria

* No repeated `users.find(...)` calls inside task loops.
* User lookup is `Map#get`.
* Scheduler output remains unchanged.

### Expected impact

For 20,000 tasks and 80 users, this removes up to roughly 1.6 million repeated comparisons per scheduler run.

## Task 2.3 — Add bounded email concurrency

### Problem

Some email sending is sequential; other paths may use unbounded parallel sends. Sequential sends are slow, while unbounded sends risk Microsoft Graph throttling.

### Files

```txt
artifacts/api-server/src/lib/reminder-scheduler.ts
artifacts/api-server/src/lib/weekly-digest-scheduler.ts
artifacts/api-server/src/routes/notify.ts
```

### New helper

```txt
artifacts/api-server/src/lib/async-utils.ts
```

### Implementation

```ts
export async function mapLimit<T, R>(
  items: T[],
  limit: number,
  fn: (item: T) => Promise<R>
): Promise<R[]> {
  const results: R[] = [];
  let index = 0;

  async function worker() {
    while (index < items.length) {
      const currentIndex = index++;
      results[currentIndex] = await fn(items[currentIndex]);
    }
  }

  await Promise.all(
    Array.from({ length: Math.min(limit, items.length) }, worker)
  );

  return results;
}
```

### Usage

```ts
await mapLimit(recipients, 5, async (recipient) => {
  await sendEmail(buildReminderEmail(recipient));
});
```

### Acceptance criteria

* Bulk email paths use bounded concurrency.
* Default concurrency should be 5 unless environment config specifies another value.
* Errors are logged per recipient without aborting unrelated sends.

### Expected impact

For 100 emails at 300 ms average Graph latency:

* Sequential: roughly 30 seconds
* Concurrency 5: roughly 6 seconds

## Task 2.4 — Stop full-tree scans for due/reminder jobs

### Problem

Reminder and digest jobs read all `clientLogs` and filter in memory. This does not scale as task volume grows.

### Required change

Add indexed mirrors for common scheduler access patterns:

```txt
tasksByDueDate/{yyyy-mm-dd}/{taskId}
tasksByAssignee/{userId}/{taskId}
```

Keep these indexes updated whenever tasks are created, updated, assigned, rescheduled, completed, or archived.

### New repository file

```txt
artifacts/api-server/src/repositories/task-repository.ts
```

### Implementation sketch

```ts
import { getAdminDatabase } from "../lib/firebase-admin";

export type TaskWrite = {
  clientId: string;
  taskId: string;
  assigneeId?: string;
  dueDate?: string;
  task: Record<string, unknown>;
};

export async function upsertTask({
  clientId,
  taskId,
  assigneeId,
  dueDate,
  task
}: TaskWrite): Promise<void> {
  const db = getAdminDatabase();

  const updates: Record<string, unknown> = {
    [`clientLogs/${clientId}/${taskId}`]: task
  };

  if (assigneeId) {
    updates[`tasksByAssignee/${assigneeId}/${taskId}`] = {
      clientId,
      dueDate: dueDate ?? null
    };
  }

  if (dueDate) {
    updates[`tasksByDueDate/${dueDate}/${taskId}`] = {
      clientId,
      assigneeId: assigneeId ?? null
    };
  }

  await db.ref().update(updates);
}
```

### Query helper

```ts
export async function getTasksDueOn(dateKey: string): Promise<Task[]> {
  const db = getAdminDatabase();

  const dueIndex = await db.ref(`tasksByDueDate/${dateKey}`).once("value");
  const refs = dueIndex.val() ?? {};

  const reads = Object.entries(refs).map(async ([taskId, ref]) => {
    const { clientId } = ref as { clientId: string };
    const snap = await db.ref(`clientLogs/${clientId}/${taskId}`).once("value");
    return snap.val() as Task | null;
  });

  return (await Promise.all(reads)).filter(Boolean) as Task[];
}
```

### Acceptance criteria

* Scheduler no longer needs to read the entire `clientLogs` tree for date-based checks.
* Task writes update primary record and indexes atomically through multi-location update.
* Existing task data is backfilled into new indexes through a migration script.

# Phase 3 — Data Model + Write Efficiency

## Task 3.1 — Convert whole-array task writes to keyed task records

### Problem

Task lists are written as whole arrays under `clientLogs/{clientId}`. This causes large payload writes and lost-update risk under concurrent edits.

### Current risky pattern

```ts
await writeFirebasePath(`clientLogs/${clientId}`, updatedLogsArray);
```

### Target data shape

```txt
clientLogs/{clientId}/{taskId} = task
```

### Required change

* Store tasks as keyed objects instead of arrays.
* Update individual task records by path.
* Use transactions for patching existing tasks.
* Use multi-location updates for indexes.

### Repository method

```ts
export async function updateTask({
  clientId,
  taskId,
  patch,
  actorId
}: {
  clientId: string;
  taskId: string;
  patch: Partial<Task>;
  actorId: string;
}) {
  const db = getAdminDatabase();

  await db.ref(`clientLogs/${clientId}/${taskId}`).transaction((current) => {
    if (!current) return current;

    return {
      ...current,
      ...patch,
      updatedAt: new Date().toISOString(),
      updatedBy: actorId
    };
  });
}
```

### Acceptance criteria

* New task writes use `clientLogs/{clientId}/{taskId}`.
* Updates patch one task instead of rewriting the whole client task list.
* Concurrent updates to different tasks do not overwrite each other.
* Migration preserves existing task IDs where available.

### Expected impact

For a client with 5,000 tasks, updating one task goes from rewriting 5,000 records to writing 1 record plus small index updates.

## Task 3.2 — Add migration script for task indexes

### New file

```txt
artifacts/api-server/scripts/backfill-task-indexes.ts
```

### Purpose

Backfill:

```txt
tasksByDueDate
tasksByAssignee
clientLogs keyed format if needed
```

### Implementation sketch

```ts
import { getAdminDatabase } from "../src/lib/firebase-admin";

async function main() {
  const db = getAdminDatabase();
  const snap = await db.ref("clientLogs").once("value");
  const clientLogs = snap.val() ?? {};

  const updates: Record<string, unknown> = {};

  for (const [clientId, tasksValue] of Object.entries(clientLogs)) {
    const tasks = Array.isArray(tasksValue)
      ? tasksValue
      : Object.values(tasksValue as Record<string, unknown>);

    for (const task of tasks as any[]) {
      if (!task) continue;

      const taskId = String(task.id ?? task.taskId ?? db.ref().push().key);
      const assigneeId = task.assigneeId ? String(task.assigneeId) : null;
      const dueDate = task.dueDate ? String(task.dueDate).slice(0, 10) : null;

      updates[`clientLogs/${clientId}/${taskId}`] = {
        ...task,
        id: taskId
      };

      if (assigneeId) {
        updates[`tasksByAssignee/${assigneeId}/${taskId}`] = {
          clientId,
          dueDate
        };
      }

      if (dueDate) {
        updates[`tasksByDueDate/${dueDate}/${taskId}`] = {
          clientId,
          assigneeId
        };
      }
    }
  }

  await db.ref().update(updates);
  console.log(`Backfilled ${Object.keys(updates).length} paths`);
}

main().catch((error) => {
  console.error(error);
  process.exit(1);
});
```

### Acceptance criteria

* Script can run idempotently.
* Script logs number of paths written.
* Script does not delete legacy data until the app is confirmed compatible.
* Add an npm script for running it manually.

## Task 5.3 — Remove API keys from query strings

### Problem

Admin trigger routes should not accept API keys in query params because query strings are often logged by proxies and hosting platforms.

### File

```txt

```

### Required change

Only accept admin trigger API keys through headers.

### Implementation

```ts
import { timingSafeEqual } from "node:crypto";

function isValidApiKey(actual: string | undefined, expected: string): boolean {
  if (!actual) return false;

  const a = Buffer.from(actual);
  const b = Buffer.from(expected);

  return a.length === b.length && timingSafeEqual(a, b);
}

function requireAdminApiKey(req, res, next) {
  const expected = process.env.ADMIN_TRIGGER_API_KEY;

  if (!expected) {
    return res.status(500).json({ error: "Admin trigger key is not configured" });
  }

  const actual = req.header("x-admin-api-key");

  if (!isValidApiKey(actual, expected)) {
    return res.status(401).json({ error: "Unauthorized" });
  }

  return next();
}
```

### Acceptance criteria

* Query-string API keys are no longer accepted.
* Header comparison uses constant-time comparison.
* Missing server-side API key fails closed.

## Task 5.4 — Replace in-memory reset-password rate limiting

### Problem

In-memory rate limiting resets when the process restarts and does not work across multiple instances.

### File

```txt
artifacts/api-server/src/routes/auth.ts
```

### Required change

Use Firebase-backed rate limit counters with expiration timestamps.

### Implementation

```ts
async function incrementRateLimit(key: string, windowMs: number) {
  const ref = getAdminDatabase().ref(`rateLimits/${key}`);
  const now = Date.now();

  const result = await ref.transaction((current) => {
    if (!current || current.expiresAt < now) {
      return { count: 1, expiresAt: now + windowMs };
    }

    return {
      count: current.count + 1,
      expiresAt: current.expiresAt
    };
  });

  return result.snapshot.val() as { count: number; expiresAt: number };
}
```

### Acceptance criteria

* Reset-password throttling works across multiple app instances.
* Counters expire after the configured window.
* Abuse attempts return `429`.

# Performance Targets

Use these targets to verify improvement:

| Area                     | Current Risk                    | Target                        |
| ------------------------ | ------------------------------- | ----------------------------- |
| Scheduler duplicate jobs | One run per app instance        | One run per schedule globally |
| User lookup in scheduler | `O(tasks × users)`              | `O(tasks + users)`            |
| Due-task lookup          | Full `clientLogs` scan          | Date-indexed lookup           |
| Single task update       | Rewrite full client task array  | Patch one task record         |
| Email sending            | Sequential or unbounded         | Bounded concurrency of 5      |
| Home view timer          | Full parent render every second | Timer component render only   |
| Group child lookup       | `O(groups × tasks)`             | `O(groups + tasks)`           |
